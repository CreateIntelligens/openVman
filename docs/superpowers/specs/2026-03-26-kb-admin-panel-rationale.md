# KB Admin 管理面板：設計與功能說明
## (2026-03-26-kb-admin-panel-rationale)

### 1. 功能概覽 (Features at a Glance)

本管理面板 (`/admin/knowledge`) 是為了打破 AI 知識管理的技術門檻而設計，讓非技術人員也能像使用 Notion 或 Google Drive 一樣維護虛擬人的「大腦」。

*   **遞迴式檔案樹 (Recursive Tree)**：完整映射伺服器端的 `workspace` 目錄，支援無限層級的資料夾管理、搬移與重新命名。
*   **智慧上傳與轉換**：使用者只需將 PDF、DOCX 或 XLSX 丟入瀏覽器，系統會自動在後端將其轉為 Markdown 格式，消除排版干擾。
*   **Markdown 雙欄編輯器**：提供「左側編輯、右側預覽」的直觀體驗，編輯器僅處理純文字，保證 AI 檢索的精準度。
*   **動態索引狀態**：透過顏色燈號實時顯示文件是否已成功載入 AI 的向量資料庫 (LanceDB)。

---

### 2. 為何這樣設計？ (Design Rationale)

#### Q1: 為什麼要將所有檔案轉為 Markdown？
*   **AI 的最佳語言**：LLM 對 Markdown 的標題、表格與結構有極佳的理解能力。直接餵 PDF 常會導致亂碼或冗餘資訊（如頁碼、頁首頁尾）。
*   **編輯便利性**：Markdown 是一種純文字格式，前端編輯器極度輕量化，且文件具備跨平台的可讀性。

#### Q2: 為什麼採用 Split-Pane（雙欄式）佈局？
*   **IDE 風格的效率**：開發者習慣在左側看樹、右側寫 code。對知識庫管理來說，這能讓用戶在不同資料夾間切換時，快速預覽並編輯內容，無需反覆進入退出頁面。
*   **上下文感知**：同時看到整體結構與單一文件內容。

#### Q3: 為什麼選擇「全自動索引」而非「手動索引按鈕」？
*   **零摩擦體驗**：我們不希望用戶去思考「什麼是 Embedding」或「什麼是向量化」。
*   **即時性**：當用戶點擊「儲存」的那一刻，後端就應該完成感知。我們在設計上將「存檔」與「重索引」綁定，讓 AI 的知識始終與最新文件同步。

#### Q4: 搜尋功能為何如此重要？
*   **海量數據與標籤**：當知識庫增長到數百份文件時，搜尋能幫助用戶快速定位到具體的「摘要」或「網址」，這也是檢視 RAG 召回準確率的第一步。
