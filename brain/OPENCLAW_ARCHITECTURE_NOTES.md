# 04_OPENCLAW_ARCHITECTURE_NOTES.md

## OpenClaw 原始碼架構與運作整理

### 1. 文件目的

本文件用來整理影片 `OpenClaw 原始碼架構與運作` 的重點，幫助快速理解 OpenClaw 的核心定位、主要流程與工程設計取向。

這份內容是根據影片摘要與解說整理而成，定位是：

- 架構理解筆記
- 二次開發參考
- 與本專案 `brain / backend` 設計對照的背景材料

不是：

- 逐檔原始碼 audit 報告
- 嚴格對照 commit 的實作證據文件

### 2. 一句話結論

OpenClaw 的本質不是聊天框，也不是單純的 LLM API wrapper，而是一個本地優先的 AI Gateway / Agent Control Plane。

它真正處理的核心問題是：

- 多通路訊息接入
- 訊息標準化與路由
- 會話上下文管理
- Prompt 組裝
- Tool Use / Agent Loop
- 容錯、重試、模型降級與穩定性治理

換句話說，OpenClaw 的價值不在「模型會回答」，而在「讓模型能在真實系統裡穩定做事」。

### 3. 核心重點整理

| 主題 | 重點 | 解讀 |
|------|------|------|
| 核心定位 | 本地優先的智慧網關（Gateway） | OpenClaw 比較像控制平面，而不是聊天 UI |
| 訊息處理 | 先標準化不同平台訊息 | 把 WhatsApp、Telegram 等來源都轉成內部統一格式 |
| Prompt 工程 | 執行環境、文件、記憶、工具說明全部進 prompt | 讓模型擁有接近「工作現場」的完整上下文 |
| Agent Loop | 不是單次回答，而是思考、用工具、觀察、再推理 | 這是它能做複雜任務的核心 |
| 工具調度 | 支援讀檔、改檔、跑命令等工具 | 模型可以不只回答，還能執行工作流 |
| 穩定性 | 內建 retry、fallback、多 key 輪詢、記憶壓縮 | 這些是面向生產環境的工程補強 |
| 代價 | Prompt 很重、token 消耗高、上下文過長可能幻覺 | 強能力換來高成本與高複雜度 |

### 4. 系統整體理解

可以把 OpenClaw 理解成三層：

#### 4.1 接入層

負責把外部世界的訊息接進來：

- 接通訊平台
- 監聽事件
- 去重
- 合併高頻訊息
- 下載附件
- 做初步權限與合法性檢查

這一層解決的是「怎麼把外部訊息穩定送進系統」。

#### 4.2 控制層

這是 OpenClaw 最重要的部分。它會負責：

- 建立 session
- 決定哪個 agent 接手
- 準備工作區與上下文
- 管理任務生命週期
- 控制工具調用
- 做錯誤處理與回退

這一層解決的是「怎麼把訊息轉成可執行任務」。

#### 4.3 推理層

這一層才是實際呼叫模型：

- 組 prompt
- 啟動 Agent Loop
- 接收 streaming 回覆
- 執行工具
- 繼續推理直到任務完成

這一層解決的是「怎麼讓模型在上下文中持續完成任務」。

### 5. OpenClaw 端到端流程

| 階段 | 核心功能 | 關鍵模組 / 檔案 | 說明 |
|------|------|------|------|
| 1. 網關啟動 | 初始化執行環境 | `automonitor.ts` | 載入配置、檢查網路、啟動各渠道監聽 |
| 2. 訊息接入 | 預處理與去重 | `inbound-monitor.ts` | 避免重複訊息與高頻抖動，把媒體與權限先處理好 |
| 3. 訊息路由 | 任務分派 | `unmessage.ts` | 判斷哪個 Agent 應處理、建立唯一 session |
| 4. 會話初始化 | 準備上下文與工作區 | `getreply.ts` | 載入歷史、初始化獨立工作區、解析特殊命令 |
| 5. 執行準備 | 打包執行參數 | `getrun.ts` | 將會話上下文、模型設定、工具環境整理成可執行配置 |
| 6. 生命週期管理 | 調度與記憶治理 | `agentrunner.ts` | 管理隊列、壓縮上下文、保存持久化記憶 |
| 7. 執行與容錯 | 失敗保護 | `execution.ts` | 處理 retry、fallback、不同執行模式切換 |
| 8. 核心推理 | 啟動 Agent Loop | `runattempts.ts` | 正式對模型發請求，進入思考/行動/觀察循環 |
| 9. 回覆分發 | 結果輸出 | `processmessage.ts` | 將 streaming 結果推回原始渠道 |

### 6. 各階段的實際意義

#### 6.1 為什麼要先 Normalize

不同通路的訊息格式差很多。如果沒有先標準化：

- 後續 prompt 組裝會變複雜
- 工具路由會耦合不同平台
- session / memory 邏輯會變亂

所以 OpenClaw 先把所有輸入轉成一種內部 message object，這是整套系統可擴充的基礎。

#### 6.2 為什麼 Prompt 會這麼重

OpenClaw 明顯採用「把執行現場儘量帶進模型」的策略。它不只給模型一句 user message，而是補上：

- 文件
- 記憶
- 工具描述
- 會話歷史
- 執行環境

這樣的好處是模型比較知道自己身處什麼任務空間，也更能正確使用工具。

代價是：

- token 成本高
- latency 變長
- 上下文太長時，可能增加幻覺風險

#### 6.3 為什麼 Agent Loop 很重要

一般聊天系統的流程是：

`輸入 -> 模型回答 -> 結束`

OpenClaw 的流程則是：

`輸入 -> 模型思考 -> 叫工具 -> 讀結果 -> 再思考 -> 再行動 -> 完成`

所以它不是單回合聊天器，而是具備執行力的任務代理層。

### 7. OpenClaw 為什麼能處理複雜任務

影片中的核心答案可以整理成三點：

#### 7.1 它把髒活累活包掉了

OpenClaw 本身不提供「智力」，而是提供一套讓智力可用的工程骨架，包括：

- 驗證
- 工作區管理
- 提示詞上下文裝配
- 工具調度
- 異常處理
- 狀態保存

#### 7.2 它讓模型不只是回答，而是能工作

只要模型可以進入 Agent Loop，它就能：

- 讀檔
- 寫檔
- 跑指令
- 根據結果修正下一步

所以能力看起來像是「在做事」，不是只是在講話。

#### 7.3 它有生產級穩定性思維

如果沒有：

- retry
- fallback
- key pool
- memory compression
- session persistence

這類系統通常一遇到真實流量或外部 API 波動就會不穩。

OpenClaw 之所以看起來成熟，是因為它先處理了這些工程現實。

### 8. 優勢與代價

| 面向 | 優勢 | 代價 / 風險 |
|------|------|------|
| 架構 | 模組清楚、分層明確 | 系統理解成本高 |
| 訊息處理 | 多平台接入友善 | 前處理邏輯會變多 |
| Prompt | 上下文完整、能力強 | token 消耗大 |
| Agent Loop | 可處理複雜任務 | 執行鏈長，debug 較難 |
| Tool Use | 可與真實系統互動 | 需做好權限與安全控管 |
| 容錯 | 比單純 chatbot 更穩 | router / fallback 設計更複雜 |

### 9. 對本專案的啟示

如果把 OpenClaw 當作參考架構，對 `openVman` 目前最直接的啟示是：

#### 9.1 `brain` 不應只停留在 RAG + generate

OpenClaw 的強點不在單次生成，而在：

- message handling
- routing
- tool orchestration
- fallback governance

這表示 `brain` 若要進一步接近可用產品，需要逐步補上這些中間層能力。

#### 9.2 workspace 是對的，但還不夠

我們現在已有：

- `SOUL.md`
- `AGENTS.md`
- `TOOLS.md`
- `MEMORY.md`
- `.learnings`

這方向是對的。但 OpenClaw 類型系統還會再往前加：

- message normalization
- execution routing
- structured tool loop
- model/key fallback router

#### 9.3 真正難的不是生成，而是治理

從影片可以看出，成熟 Agent 系統最難的不是把模型叫起來，而是：

- session 怎麼管
- context 怎麼壓
- 工具怎麼調
- 失敗怎麼補
- 回覆怎麼穩定輸出

這些正是產品化時最容易被低估的部分。

### 10. 建議的閱讀方式

如果後續要用這份整理去推動實作，建議順序是：

1. 先把 OpenClaw 當成「控制平面」理解，不要只把它看成聊天器
2. 先拆清楚接入層、控制層、推理層
3. 優先學它的訊息標準化、Agent Loop、fallback 機制
4. 不要照抄整套 prompt 重裝策略，先評估 token 成本與必要性

### 11. 最終總結

OpenClaw 的核心價值可以濃縮成一句話：

它不是在包模型，而是在包模型工作時所需要的整個操作系統。

它之所以強，不是因為用了什麼神奇模型，而是因為把下列能力做成了一條完整流水線：

- 訊息接入
- 格式標準化
- 上下文管理
- Prompt 組裝
- 工具執行
- 容錯與回退
- 結果分發

如果目標是做出「能實際工作的 AI 系統」，OpenClaw 的參考價值主要不在 prompt 文案，而在這套控制平面與任務執行架構。
